Use the current Faultline Lab codebase, case engine, product catalog, entitlement system, store UI, and commercial productization plan as the source of truth. Do NOT restart from scratch. Do NOT create a second disconnected case catalog system.

MISSION
Implement a structured, typed, reusable case catalog and pack-mapping layer so content packs, starter access, premium locks, product detail pages, Case Files browsing, and entitlement-aware case launching all use one clean source of truth.

This task is specifically to connect the commercial catalog to the actual case library.

The result must ensure:
- each pack points to real case IDs
- each case knows which product/pack unlocks it
- starter/free cases are clearly distinguished
- locked/unlocked case state is derived from entitlements
- Case Files and store product detail views can both render from the same metadata layer
- no paid pack appears to include content that cannot actually launch

DO NOT:
- hardcode case metadata in UI components
- duplicate case definitions across multiple unrelated files
- create fake “included cases” lists in store cards
- leave paid pack case cards pointing to missing implementations
- change existing case IDs if they already exist and are used safely unless you update all references carefully
- create a second parallel registry that competes with the existing product catalog
- break billing, entitlement, or current working case-launch behavior

DO:
- preserve working architecture
- centralize case catalog data
- use stable case IDs and slugs
- connect cases to real products and entitlements
- make case counts derive from actual mappings
- validate the mapping layer
- keep the system extensible for future packs and specialty bundles

PREFERRED IMPLEMENTATION APPROACH
Use TypeScript seed/config files with strong typing. Keep the structure JSON-like and data-driven so it could later move to a database or CMS.
