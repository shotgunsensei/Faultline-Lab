PROJECT: Faultline Lab
ROLE: Build engineer and product implementer
DELIVERABLE: A genuinely playable MVP of Faultline Lab, not a mockup

PROJECT SUMMARY
Build "Faultline Lab", a cinematic browser-based troubleshooting simulator for technical minds. Users investigate broken systems across IT infrastructure, networking, Windows/Active Directory, servers, automotive diagnostics, and smart electronics. The experience must be an interactive simulation driven by evidence, tool use, state changes, and diagnosis logic. It must not behave like a quiz or static dashboard.

PRIMARY OBJECTIVE
Deliver a working MVP with a complete end-to-end investigation loop:
1. Launch app
2. Enter Incident Board
3. Select a case
4. Investigate with working tools
5. Unlock evidence through actual interactions
6. Submit diagnosis
7. Receive score and debrief
8. Persist progress in browser storage

NON-NEGOTIABLE RULES
- Do not build a mostly static concept UI
- Do not create dead buttons, fake panels, or placeholder-only screens
- Do not hardcode case logic directly inside UI components
- Do not prioritize animation over function
- Do not reduce the app to trivia or multiple choice
- Every major interaction must be backed by real state and real logic
- Evidence must unlock through actual tool usage or investigation milestones
- Diagnosis must be evaluated against the actual hidden case data
- State must persist on refresh via localStorage
- Build the playable core first, then expand and polish

TECH STACK
Use:
- React
- TypeScript
- Vite
- Tailwind CSS
- Zustand
- Framer Motion
- Recharts
- localStorage for persistence

OUT OF SCOPE
- Real backend or database
- Multiplayer or collaboration
- Audio assets beyond a settings toggle placeholder
- Deep mobile optimization
- External APIs unless absolutely required

DESIGN DIRECTION
Desktop-first. Premium dark cyber-diagnostic-lab interface.
Target feel:
- high contrast dark surfaces
- glowing status accents
- structured modular panels
- clean typography and spacing
- terminal-style outputs
- restrained animation
- looks like a real high-end diagnostic console, not a game jam prototype

CORE PRODUCT MODES
Target end state includes:
- Case Files
- Random Incident Generator
- Sandbox Builder
- Daily Challenge
- Free Investigation

IMPORTANT:
For MVP, implement only what is required for a complete playable loop first.
Do not build all modes in parallel.

================================================================================
PHASE 2: MVP PLAYABLE LOOP
================================================================================

GOAL
Build one complete investigation workflow before expanding the product.

REQUIRED MVP MODES
- Incident Board
- Case Files only

REQUIRED MVP CASES
Build 4 fully playable handcrafted cases first:

1. Windows / Active Directory
Scenario:
A domain-joined PC cannot authenticate after a password reset.
Potential clue areas:
- cached credentials
- domain controller reachability
- DNS resolution
- time skew
- policy or trust confusion
Root cause must be logical and discoverable.

2. Networking / VPN
Scenario:
A site-to-site VPN shows green/up but traffic between subnets is failing.
Potential clue areas:
- routing
- firewall policy
- NAT
- phase/policy mismatch
- asymmetric path behavior
The visible tunnel health should function as a believable red herring.

3. Automotive Diagnostics
Scenario:
A vehicle has unstable idle, intermittent misfire counts, and voltage fluctuation behavior.
Potential clue areas:
- OBD live data
- RPM fluctuation
- voltage behavior
- trims
- misfire counts
- sensor or component observations
Root cause must be logically solvable through data.

4. Electronics / Sensor Mesh
Scenario:
A smart sensor network has one unstable node causing mesh poisoning or false telemetry.
Potential clue areas:
- voltage irregularity
- signal quality
- node logs
- firmware mismatch
- topology behavior
Root cause must be discoverable and fair.

REQUIRED MVP WORKSPACE
Build the investigation workspace with:
- header showing case title, category, difficulty, and status
- tool navigation
- main active tool panel
- sidebar or secondary panels for:
  - symptoms summary
  - action log
  - evidence locker
  - hint controls
- diagnosis submission entry point
- save/exit support

REQUIRED MVP TOOLS
Implement these first and make them functional:

1. Terminal Panel
Commands should be case-aware and return outputs that reflect real case state.
Support at minimum:
- ping
- tracert
- nslookup
- ipconfig
- dcdiag
- route print or equivalent
- basic service or adapter status checks

2. Event Log Reader
- Show case-relevant logs
- Include clue-bearing warnings/errors/events
- Different cases should surface different log streams or event sets

3. Ticket History / User Statement Panel
- Original issue description
- user observations
- prior technician notes
- believable red herrings where appropriate

4. Evidence Locker
- Evidence appears only when unlocked through actual interactions
- Collected evidence must persist in case state

5. Diagnosis Submission Form
Require:
- suspected root cause
- supporting evidence
- proposed remediation

REQUIRED MVP SUPPORT SYSTEMS
- Action log
- 4-tier hint system
- Scoring
- Debrief screen
- localStorage persistence

HINT SYSTEM
Implement four hint levels:
1. subtle nudge
2. directional clue
3. stronger clue
4. reveal path

Hints must be case-specific and should reduce maximum score.

SCORING SYSTEM
Score based on:
- diagnosis accuracy
- evidence quality
- remediation quality
- efficiency
- unnecessary risky actions
- excessive random probing if applicable
- hint penalties

Suggested result tiers:
- Surgical
- Solid
- Sloppy but Correct
- Misdiagnosed

DEBRIEF REQUIREMENTS
Show:
- actual root cause
- clues that mattered
- misleading clues
- recommended remediation
- preventative measures
- score breakdown
- achievements unlocked if any

PHASE 2 ACCEPTANCE CRITERIA
- User can open the app, select a case, investigate, collect evidence, submit a diagnosis, receive a score, and review a debrief
- All 4 MVP cases are playable end to end
- Tool outputs are case-aware
- Evidence unlock logic works
- Hints work
- Scoring works
- Debrief works
- State persists on refresh
- No placeholder-only panels in the MVP loop
